RS-02 - Ingest real GFS Wave 1/2 non-sensitive data and implement flourishing scoring validation.

Objective: Ingest real GFS Wave 1/2 non-sensitive data where available and implement the six-domain HFS/GFS flourishing scoring kernel with longitudinal validation tables.

Implement deliverables:
- GFS connector with COS/OSF acquisition pattern, access-mode detection, and sensitive/non-sensitive split
- codebooks/gfs_wave1.yaml and codebooks/gfs_wave2.yaml with source labels preserved
- Scoring kernel for 12-item Secure Flourish and 10-item Flourish indices with weights
- features.gfs_wave12_panel_non_sensitive and features.gfs_wave12_marginals_country_*
- Dataset cards documenting access, weights, limitations, and citation instructions
- Request-status records for Wave 3/sensitive data rather than fabricated microdata

Acceptance gates:
- GFS Wave 1/2 non-sensitive connector can register real data when access is available and otherwise creates request-status stubs
- Scoring tests reproduce known country/domain aggregate checks from source documentation within tolerance
- Sensitive variables are inaccessible without approved access policy
- The Atlas and run manifests label GFS Wave 1/2 as real data and any Sweden/sample data as sample/perturbed only

Constraints:
- Do not fabricate data. Fixtures are allowed only for unit tests and must be named fixture_only.
- Every production-facing dataset must have a dataset card, license/access metadata, codebook mapping, quality profile, provenance manifest, and content hash.
- Every simulation-facing output must use dataset_reference = (canonical_dataset_name, version, content_hash).
- MiroFish outputs are interaction/narrative artifacts unless explicitly validated; they must not create causal effect sizes.
- Keep changes scoped to this sprint and do not commit automatically.
